# asana.md 
Asanaのタスク概要欄に入れる説明文を、Claudeに作成してもらうためのテンプレート。
誰が読んでも理解できる形にするために使う。

---

## Asanaタスク階層構造

```
 [親タスク] Claudeを用いたWeb制作プロセス標準化検証（-test_sct）
   │  → プロジェクト全体の目的・概要・参照リポジトリを記載
   │
   ├─ [サブタスク] 0-1 全体｜プロセス履歴管理を記録する
   │  ├─ プロジェクト共通ルールを定義するためにCLAUDE.md を作成
   │  ├─ プロセス履歴を記録するためにgeneral-log.md を作成（Keep a Changelog形式）
   │  ├─ mdの履歴を管理するためにGitリポジトリを作成
   │  ├─ 議事録を蓄積するためにmeeting-notes.md を作成

   │
   ├─ [サブタスク]  1-1 企画・設計｜要件を定義
   │  ├─ 検証用サイトのテーマ・目的をClaudeと壁打ちで決定
   │  ├─ ゴール・KPIを明確にするために箇条書きで整理
   │  ├─ 要件を整理するためにrequirements.md を生成・保存
   │  └─ requirements.md を共有しやすいようExcelに変換
   │
   ├─ [サブタスク]  1-2 企画・設計｜サイトマップを作成
   │  ├─ ページ構成を整理するためにClaudeに出してもらう
   │  ├─ サイト構造を整理するためにsitemap.md を作成
   │  └─ sitemap.md を共有しやすいようExcelに変換
   │
   ├─ [サブタスク]  1-3 企画・設計｜競合分析・ペルソナを元にしたデザインコンセプトを策定
   │  ├─ ブランドの世界観を具体化するために「ブランドの世界にあるモノ」を5個以上書き出す ★AI感排除
   │  ├─ 避けたい方向性を明確にするために「絶対に見えたくないサイト」を1件収集 ★AI感排除
   │  ├─ 競合を分析するためにcompetitor.md を作成
   │  ├─ ターゲットを明確にするためにuser-persona.md を作成
   │  ├─ 行動導線を整理するためにcustomer-journey.md を作成
   │  └─ デザイン方針を決めるためにdesign-concept.md を作成
   │
   ├─ [サブタスク]  1-4 企画・設計｜スケジュールを立てる
   │  ├─ 進行計画を決めるために1ヶ月のマイルストーンをClaudeと設定

   │  ├─ Asana連携（MCP経由）が使えるか確認
   │  └─ schedule.md をAsanaへタスク登録
   │
   ├─ [サブタスク]  1-5 企画・設計｜クライアント向けの提案書を作成
   │  ├─ proposal.md の構成案をClaudeに作らせる
   │  ├─ proposal.md をPPTXに変換
   │  └─ PPTXをPDFに書き出し
   │
   ├─ [サブタスク]  1-6 企画・設計｜コンポーネント・DB/API設計の仕様書を作成
   │  ├─ CMS管理対象を整理するためにコンテンツを洗い出す
   │  ├─ UI仕様を定義するためにcomponents.md を作成
   │  ├─ 機能仕様を定義するためにspec.md を作成
   │  ├─ DB設計を定義するためにschema.md を作成
   │  ├─ API仕様を定義するためにapi.md を作成
   │  ├─ 技術選定理由を整理するためにarchitecture.md を作成
   │  └─ 仕様書を共有するためにPPTX/PDFへ変換
   │
   ├─ [サブタスク]  1-7 企画・設計｜デザインコンセプトと仕様書を元にしたワイヤーフレームを策定
   │  ├─ Figmaで新規プロジェクトファイルを作成
   │  ├─ Claude Codeから操作できるようFigma MCP接続を確認
   │  ├─ TOPページのワイヤーフレームをFigma MCP経由で生成
   │  ├─ 下層ページも同様に生成
   │  ├─ CMSも同様に生成
   │  └─ 設計意図を残すためにwireframe.md に記録
   │
   ├─ [サブタスク]  2-1 デザイン｜ワイヤーフレームを元にしたデザイン案を作成
   │  ├─ 比較検討するために3方向（保守的／実験的／中間）を提案させる ★AI感排除
   │  ├─ 最適案を選ぶために人間が1つを選定（レビューゲート①） ★AI感排除
   │  ├─ 採用するデザイン案を決定
   │  ├─ デザインルールを確定するためにdesign-tokens.md を作成 ★AI感排除

   │  ├─ 下層ページを作成
   │  ├─ CMSページを作成
   │  └─ デザインルールをまとめるためにdesign-system.md を作成
   │
   ├─ [サブタスク]  2-2 デザイン｜コンセプト・仕様書との整合性チェック
   │  ├─ design-concept.md とFigma案の照合項目を記載
   │  ├─ チェック結果を反映・修正
   │  ├─ 定番パターンや不要な装飾を洗い出すためAI自己批評を実施 ★AI感排除
   │  ├─ AI視覚レビューを実施（デスクトップ＋モバイル） ★AI感排除
   │  ├─ AI感検出8項目を全てクリアしたことを確認 ★AI感排除
   │  ├─ 指摘事項を反映
   │  └─ 採用理由を残すためにdesign-log.md に記録
   │
   ├─ [サブタスク]  3-1 構築・開発｜Claude Codeを用いたフロントを実装
   │  ├─ Next.jsプロジェクトを新規作成
   │  ├─ Figma MCP経由でTOPページをClaude Codeに生成させる
   │  ├─ AI感を抑えるため段階生成で実装する ★AI感排除
   │  ├─ デザインルールを反映するため各生成前にdesign-tokens.md を添付 ★AI感排除
   │  ├─ レスポンシブ対応を確認
   │  └─ 下層ページも同様に実装
   │
   ├─ [サブタスク]  3-2 構築・開発｜CMS・API実装
   │  ├─ SQLセットアップ方法を把握
   │  ├─ schema.md を元にDB作成方法を把握
   │  ├─ APIエンドポイントを把握
   │  ├─ 管理画面UIの実装方法を把握
   │  └─ フロントとAPIの接続方法を把握
   │
   ├─ [サブタスク]  3-3 構築・開発｜コードレビューと決定ログの記録
   │  ├─ Claudeとコードレビューを実施
   │  ├─ 開発ルールをまとめるためにcontributing.md を作成
   │  ├─ 用語対応を整理するためにglossary.md を作成
   │  └─ 実装判断を残すためにdevelopment-log.md に記録
   │
   ├─ [サブタスク]  4-1 検証・公開｜デザインと整合性チェック・動作チェック
   │  ├─ AI感検出チェックを検証項目に追加 ★AI感排除
   │  ├─ Claude経由でチェックリストを作成
   │  ├─ Figmaデザインとの整合性を確認
   │  ├─ 全体動作を確認
   │  ├─ 不具合を修正
   │  └─ デプロイ先を決めて公開手順を把握
   │
   ├─ [サブタスク]  4-2 検証・公開｜運用マニュアル作成
   │  ├─ 運用手順をまとめるためにmanual.md を作成
   │  ├─ クライアント向けCMS操作手順を記載
   │  └─ Wordへ変換し体裁を確認
   │
   └─ [サブタスク]  5-1 運用｜アクセスを分析と決定ログの記録
      ├─ GA4を設定
      ├─ 改善を継続するため分析サイクルを作成

      └─ 判断理由を残すためにoperations-log.md に記録
```

## 使い方

1. 作りたいタスクの階層（親／サブタスク1／サブタスク2）を選ぶ
2. 該当するプロンプトの各項目を埋める
3. Claudeに貼り付けて実行する
4. 出力結果をAsanaの概要欄にそのまま貼る

---

## 親タスク用プロンプト

```
以下の情報をAsanaの親タスク概要として作成してください。

【プロジェクト名】
[プロジェクト名を入力]

【プロジェクトの目的】
[このプロジェクトが存在する理由・解決する問題を入力]

【含まれるフェーズ】
[0.全体管理 / 1.企画・設計 / 2.デザイン / 3.構築・開発 / 4.検証・公開 / 5.運用]

【参照リポジトリ】
[GitHubのURLを入力]

条件：
- 専門知識がない人でも理解できる日本語で書く
- プロジェクト全体を俯瞰できる内容にする
- 全体で200字以内に収める
```

## サブタスク1（プロセス）用プロンプト

```
以下の情報をAsanaのサブタスク1（フェーズ）概要として作成してください。

【フェーズ名】
[例：1. 企画・設計]

【このフェーズの目的】
[このフェーズで何を達成するかを入力]

【含まれるタスク】
[例：1-1 要件を定義 / 1-2 サイトマップを作成 / ...]

条件：
- 専門知識がない人でも理解できる日本語で書く
- このフェーズが終わった時に何が揃っているかが分かる内容にする
- 全体で150字以内に収める
```

## サブタスク2（プロセスのタスク）用プロンプト

```
以下の情報をAsanaのサブタスク2（個別タスク）概要として作成してください。

【タスク名】
[例：1-1 企画・設計｜要件を定義]

【なぜやるのか（目的）】
[このタスクが存在する理由・解決する問題を入力]

【用語解説】
[タスク名や説明に出てくる専門用語があれば入力。なければ「なし」]

【完了条件】
[「これができたら完了」という具体的な基準を入力]

【参照先】
[関連するmdファイルのパスやURLがあれば入力。なければ「なし」]

条件：
- 専門知識がない人でも理解できる日本語で書く
- 用語は初出時に必ず一言説明を入れる
- 「なぜやるのか」は必ず最初に書く
- 全体で300字以内に収める
```

## 出力例

### 親タスクの例
```
【目的】
Claude CodeとFigma MCPを活用し、デザイナー一人で企画から運用まで
完結できるWeb制作プロセスを検証するプロジェクト。

【フェーズ構成】
0. 全体管理 → 1. 企画・設計 → 2. デザイン → 3. 構築・開発 → 4. 検証・公開 → 5. 運用

【参照】
https://github.com/yoouuttthhh/-test_sct
```

### サブタスク1の例（1. 企画・設計）
```
【目的】
サイトの要件・構成・デザイン方針・スケジュールを固め、
クライアントへの提案と開発チームへの仕様共有を完了させるフェーズ。

【含まれるタスク】
・1-1 要件を定義
・1-2 サイトマップを作成
・1-3 デザインコンセプトを策定
・1-4 スケジュールを立てる
・1-5 提案書を作成
・1-6 仕様書を作成
・1-7 ワイヤーフレームを策定
```

### サブタスク2の例（0-1 プロセス履歴管理を記録する）
```
【目的】
AIと共同開発するうえで「何をいつ変えたか」を記録し、
Claudeがプロジェクトの文脈を瞬時に把握できる状態を作るため。
記録がないとAIは過去の判断を忘れ、同じ議論を繰り返す原因になる。

【用語】
・CLAUDE.md：Claudeがリポジトリを理解するための指示書
・CHANGELOG.md：変更履歴をAdded/Changed/Removed/Fixedで記録するファイル
・Gitリポジトリ：ファイルの変更履歴を管理する仕組み

【完了条件】
・CLAUDE.md が作成されている
・CHANGELOG.md が作成されKeep a Changelog形式で記録されている
・Gitリポジトリが作成されている

【参照】
https://github.com/yoouuttthhh/-test_sct
```
