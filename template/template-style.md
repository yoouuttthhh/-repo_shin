# template-style.md
mdを新規作成する時のスタイルを参照するテンプレート。
mdの具体的な構成は [CLAUDE.md](./CLAUDE.md) に従うこと。


## 制約事項・禁止事項
- 外部リンクの時にURLをそのまま貼り付けることをやめる
　例）https://b-risk.jp/blog/2023/10/banner-design/
- リンク先（URL・ファイルパス）が判明しているテキストは、可能な限りMarkdownリンクにする
　例）[製品パンフレット(623KB)（PDF）](URL)
- 変更理由・補足の説明文は1箇所にのみ記載し、更新履歴と一覧表など複数箇所に重複して書かない
- 空欄可の項目（備考など）は、明記する指示がなければ空欄にする（推測で埋めない）
- 該当する内容がない見出し・セクションは省略してよい（空のセクションを残さない）


## 出力形式

### 見出し
- 一番最初の1行は `# ファイル名.拡張子` にする（H1の扱い）
- H2は `##` で表示する
- H2の下層は `###`
- H3の下層は `**内容**`
- 情報の下層が足りない場合は `###` などを省略してもOK
- `##` には `1.` `2.` などの番号をつけない
- 見出し（`##`/`###`）配下で解説する項目が複数あり、各項目に補足説明・事例・対象者などの情報が多く複雑な場合は、下記「番号付き詳細リスト」の形式にしてよい


### 番号付き詳細リスト（見出し配下の情報が複雑・多い場合）
- 項目は `1.` `2.` の番号付きリストにする
- 各項目の先頭は `**項目名**` で太字にする
- 各項目の下は `-` の箇条書きで詳細（説明・事例・対象者など）を並べる
- 1行に収まらない長い文はスペースでインデントし、箇条書きの位置に折り返しを揃える

**番号付き詳細リストの例**
```
**AI活用の3段階（チャット型 → Claude Cowork → Claude Code）**

1. **チャット型AI（ChatGPT・Claude等）**
   - 自然言語でやり取りし、文章作成・観点整理・構成案の提示などができる
   - ただし出力は「答えを返すだけ」で、資料への転記・体裁調整・保存などの
     実作業は人間が行う必要がある

2. **Claude Cowork（コアーク）**
   - 指示に対して作業の段取りを自分で立て、Web調査や資料参照を行い、
     Excel/PowerPoint等のファイルとして成果物を仕上げるところまで自動で行う
   - 人間は作業中に別のことをしていられ、完了通知を受けて確認するだけでよい
   - 事例：週3時間かかっていた競合調査業務が、依頼10分＋確認15分の25分に短縮
     （約80%超の時間削減）
   - 向いている人：自分一人で使う、その都度AIと相談しながら進めたい人

3. **Claude Code（コード）**
   - 作業手順を「型」として固定し、ボタン一つで毎回同じ処理を実行できる
     アプリケーションとして開発できる
   - チャットでのやり取りすら不要になり、自分以外の人にもツールとして渡せる
   - 事例：競合ウォッチャーのような、チェックボタン一つで調査を実行する
     デスクトップツールの開発
   - 向いている人：定型作業を繰り返し行う、他者にもツールとして展開したい人
```


### リスト・テキスト
- `# ファイル名.拡張子` の下には目標を２行以内で記載（pスタイルの扱い）
- リストでまとめるときは `-` を使う
- ファイル・ディレクトリ・コマンドは `` ` `` で囲む
- 外部リンクは `> 出典・関連ファイル・関連リンク [リンク名](./CLAUDE.md)` 形式にする
　例）> 出典 [CLAUDE.md](./CLAUDE.md)

### 区切り・間隔
- `##` 単位の間は2回改行する
- `---` は禁止


### 時系列（更新履歴・議事録など）
- 1エントリーにつき `##` 見出しを1つ使う
- 見出しは `## [YYYY-MM-DD]` 形式にする（日付はISO 8601、`[ ]` で囲む）
- タイトルを付ける場合は `## [YYYY-MM-DD]　タイトル` のように全角スペースで区切って見出しに続ける
- 新しいエントリーほどファイルの先頭にくるように並べる（逆時系列）
- 未確定・進行中の内容をまとめる場合は `## [Unreleased]` をファイル最上部に置く
- エントリー内をカテゴリ分けする場合は `###` を使う（例：`Added`/`Changed`、`決定事項`/`課題` など）
- `###` の下は箇条書き（`- `）で記載する。「・」など記号を先頭につけただけの疑似箇条書きは使わない（プレビュー表示時に改行されないため）
- 箇条書きの中でラベルを強調する場合は `**ラベル**` を使う（`###`見出しにはしない）
- 変更履歴として `### Added` / `### Changed` / `### Removed` を使う場合は、各項目を `- **項目名**：理由` の形式で書く

### バージョン管理（日付だけでなく版数も管理したい場合）
- 見出しを `## [ver.X.X（YYYY-MM-DD）]` 形式にする（`[ ]` で囲む）
- 初版は `ver.0.1` とする
- 通常の追加・修正・削除は `0.1` ずつ増やす（例：ver.0.1 → ver.0.2 → ver.0.3）
- 大規模な刷新（リニューアル等）の場合は整数を繰り上げる（例：ver.0.9 → ver.1.0）
- 参考：[sitemap-tem.md](./Sitemap/sitemap-tem.md)

### スナップショット型の更新履歴（各バージョンの内容を完全な形で残したい場合）
サイトマップなど、差分だけでなく「そのバージョン時点の全体像」自体を毎回残しておきたいファイルに使う。
- 各バージョンの見出し配下に、`### Added`/`### Changed`/`### Removed`（変更差分）と、その時点の完全な内容（ツリー・一覧表など）の両方を記載する
- 差分があった項目には、ツリー・一覧表側にも `★Added` / `★Changed` / `★Removed` のタグを付けて可視化する
- 過去バージョンのブロックには差分タグを付けず、その時点の内容をそのまま記録する（新しいバージョンほど上に置く逆時系列）
- 参考：[sitemap-tem.md](./Sitemap/sitemap-tem.md)

**時系列の例**
```
## [Unreleased]

## [2026-07-22]　タイトル（任意）
概要を1行で記載

### カテゴリA
- **ラベル**：内容

### カテゴリB
- **ラベル**：内容
```

### テーブル
- コマンド集などの場合はテーブル化する

| やること | コマンド | 説明 |
|---|---|---|
| フォルダーに移動する | `cd /path/to/your-repo` | 指定したフォルダーに移動する |
| VS Codeを現在フォルダーに切り替える | `code -r .` | 今開いているウィンドウをこのフォルダーに切り替える |
| VS Codeを新しいウィンドウで開く | `code .` | 新しいウィンドウでVS Codeを開く |
